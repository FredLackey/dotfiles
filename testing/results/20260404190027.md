## Errors

### Error 1: Claude Code PATH verification failure
- **Phase**: claude-code.sh (AI category)
- **Exit Code**: 1
- **Error Message**:
```
⚠ Setup notes:
  • Native installation exists but ~/.local/bin is not in your PATH. Run:

  echo 'export PATH="$HOME/.local/bin:$PATH"' >> your shell config file &&
  source your shell config file

✔ Claude Code successfully installed!

  Version: 2.1.92

  Location: ~/.local/bin/claude

...

Error: Claude Code installation failed.
```
- **Root Cause**: Claude Code's own installer (`curl -fsSL https://claude.ai/install.sh | bash`) places the binary at `~/.local/bin/claude`. The dotfiles `shell-config.sh` installer runs before this point and deploys `.bashrc`/`.bash_profile`, but those files are not sourced into the currently running installer script session. As a result, `~/.local/bin` is never added to `PATH` within the subprocess running `claude-code.sh`. The post-install `command -v claude` check fails even though the binary exists at `~/.local/bin/claude`.
- **Suggested Fix**: In `claude-code.sh`, replace the post-install check with a direct file existence check. After the install step, verify that the binary file exists at its known install path rather than relying on `command -v claude` (which requires PATH to be current):

```bash
# 4. VERIFY - Check binary at known location (PATH may not be updated in current session)
CLAUDE_BIN="$HOME/.local/bin/claude"
if [ -f "$CLAUDE_BIN" ] || command -v claude >/dev/null 2>&1; then
    echo "$APP_NAME installed successfully."
else
    echo "Error: $APP_NAME installation failed."
    exit 1
fi
```

## Recommendations

1. **Fix claude-code.sh verification (blocking)**: The verify step must not rely on `command -v claude` because the installer session's PATH is not updated after shell-config.sh deploys the profile files. Use a direct file existence check against `$HOME/.local/bin/claude` as the primary check, with `command -v claude` as a fallback. Example:

```bash
CLAUDE_BIN="$HOME/.local/bin/claude"
if [ -f "$CLAUDE_BIN" ] || command -v claude >/dev/null 2>&1; then
    echo "$APP_NAME installed successfully."
else
    echo "Error: $APP_NAME installation failed."
    exit 1
fi
```

2. **PATH export in setup.sh (optional enhancement)**: After shell-config.sh runs, setup.sh could explicitly export `PATH="$HOME/.local/bin:$PATH"` so that subsequent installers benefit from the updated PATH without needing to source `.bashrc`. This would make the verification logic in claude-code.sh work with `command -v` as written, and would be consistent with how NVM is reloaded after nvm.sh.

3. **Gitego upstream (non-blocking)**: The graceful skip behavior is correct. Consider adding a comment in gitego.sh noting that arm64 binaries may not always be available upstream.

4. **kiro-cli.sh and remaining installers**: They were not reached due to the claude-code.sh failure. A pass on claude-code.sh should be confirmed before drawing conclusions about those installers.

## Troubleshooting Results

**Date**: 2026-04-04
**Environment**: ubuntu-server

### Issues Identified

1. Claude Code PATH verification failure
   - Root Cause: `claude-code.sh` used `command -v claude` for both the CHECK (already-installed guard) and the VERIFY step. The Claude Code upstream installer places the binary at `~/.local/bin/claude`. During the dotfiles installer session, `shell-config.sh` has already deployed `.bashrc` and `.bash_profile` but those files are never sourced into the currently running shell process, so `~/.local/bin` is never added to `PATH`. As a result `command -v claude` returns nothing even though the binary exists, causing a false failure on the VERIFY step and a non-zero exit that aborts the entire setup run.
   - File Modified: `src/os/ubuntu-server/installers/claude-code.sh`
   - Fix Applied: Updated both the CHECK step and the VERIFY step to use a combined guard: `command -v claude >/dev/null 2>&1 || [ -f "$HOME/.local/bin/claude" ]`. The file existence check is placed first in the VERIFY step since it is the reliable signal when PATH has not been updated. Added inline comments explaining why the file check is necessary.
