## Fix Applied During This Session

### Fix: td.sh and sidecar.sh — go.mod version mismatch (commit 355283b)

**Root cause:** The ubuntu-wsl `td.sh` and `sidecar.sh` installers used `set -e` with no error guard around `go install`. When `go install` failed due to the upstream `go.mod` requiring Go 1.25+ (Ubuntu 22.04 apt provides Go 1.18.x), the script exited with code 1, propagating a fatal failure up through `setup.sh`.

**Fix:** Ported the graceful-skip pattern from ubuntu-server to ubuntu-wsl for both installers. The pattern wraps `go install` in a conditional block and exits 0 with a warning message if installation fails, matching how `gitego.sh` already handled the same class of error.

**Discovery context:** The ubuntu-server versions of these files were fixed in the previous session (commits d0e647d and e71dd82). Per the OS isolation rule, ubuntu-wsl is a completely independent folder and must be fixed separately — fixes to ubuntu-server do not propagate.

**First test run result (before fix):** FAIL — exit code 1, stopped at td.sh
**Second test run result (after fix):** PASS — all installers completed

- **Analysis**: NVM installs to `~/.nvm` and modifies `.bashrc`, but in-session PATH sourcing requires explicit sourcing in a non-interactive shell. The `setup.sh` sources `nvm.sh` after installation so subsequent npm-dependent installers work correctly. Cosmetic only.

### Warning 2: WSL log file skipped
- **Phase**: setup.sh initialization
- **Behavior**: The WSL log setup block calls `cmd.exe` to find the Windows USERPROFILE path. In Docker, `cmd.exe` is not available, so `WIN_USERPROFILE` is empty and `LOG_FILE` is never set. As a result, all installer output is suppressed (`>/dev/null 2>&1`) rather than routed to either a log file or the console.
- **Analysis**: This is correct behavior. In a real WSL environment, `cmd.exe` would succeed and the log would be written to the Windows Desktop. In Docker the suppression is expected. The `--------------------------------------------------` phase headers still appear because they are printed before the installer `bash` call.

### Warning 3: AWS CLI QEMU warning on ARM64
- **Phase**: aws-cli.sh
- **Analysis**: Docker host is Apple Silicon (ARM64). AWS CLI install fetches an x86_64 binary. QEMU translation lacks the x86_64 linker. The binary is placed at `/usr/local/bin/aws` and the installer reports success. Not a code defect — host architecture issue only.

- **Analysis**: `bgreenwell/gitego` upstream declares `go 1.24.4` in its `go.mod`, which exceeds Ubuntu 22.04 apt's Go 1.18.x. The installer exits 0 gracefully. Pre-existing graceful-skip pattern was already correct.

## WSL-Specific Observations

### Output Suppression Behavior
The ubuntu-wsl `setup.sh` has a two-path output model:
1. If `LOG_FILE` is set (real WSL with Windows Desktop available): verbose installer output goes to the log file; console shows phase headers only.
2. If `LOG_FILE` is not set (Docker or WSL without Windows path resolution): installer output is suppressed entirely with `>/dev/null 2>&1`; console shows only phase headers.

This means the Docker test output shows installer phase headers but no per-tool success/failure messages. This is expected behavior, not a defect. On a real Windows/WSL machine the log file on the Desktop would contain the full details.

### wsl.conf Written Successfully
The `preferences/wsl.sh` script wrote `/etc/wsl.conf` with interop settings and the current user as the default. In Docker this file is written but has no operational effect. On a real WSL machine this file controls WSL behavior after `wsl --shutdown`.

### Differences from ubuntu-server
- ubuntu-wsl includes `starship.sh` and `starship-config.sh` (ubuntu-server does not)
- ubuntu-wsl does not include `docker.sh`, `tailscale.sh`, or `nightshift.sh` (ubuntu-server does)
- ubuntu-wsl preferences are `locale.sh`, `git.sh`, `wsl.sh` (ubuntu-server has `system.sh`, `git.sh`, `ssh.sh`)
- ubuntu-wsl installer output is suppressed in Docker; ubuntu-server shows all output

## Recommendations

### Upstream Go Compatibility (td, sidecar, gitego)
Same root cause as ubuntu-server: Ubuntu 22.04 apt Go (1.18.x) is too old for all three upstream `go.mod` declarations. The graceful-skip approach is the correct short-term fix. Long-term options:
1. Install Go from the official tarball (golang.org) to get a current version (1.21+).
2. Upgrade the base Ubuntu image to 24.04.
3. Accept the skips as permanent and document these tools as needing manual Go upgrade.

### Consider Verbose Pass/Fail Summaries in Docker
Since installer output is suppressed when no LOG_FILE is set, a test failure in Docker produces no detail about which specific command inside an installer failed. The phase header (`Running installer: X.sh`) is the last visible line before the failure. Consider adding a fallback that logs to a temp file in `/tmp` when `LOG_FILE` is empty, so Docker test runs still capture verbose output for debugging.

### AWS CLI on ARM64
Same QEMU warning as ubuntu-server. Not actionable — host architecture issue only. Disappears on real AMD64 hardware.
